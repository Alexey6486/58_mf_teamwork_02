## Методика проверки
Проверка выполнена статическим анализом кода по типовым источникам утечек:
- незакрытые таймеры (`setInterval`, `setTimeout`)
- забытые подписки/листенеры (`addEventListener`, `store.subscribe`, сокеты)
- неотменённые async-операции (fetch/AbortController в SSR)
- незакрытые серверные ресурсы (HTTP server, DB connections)

Поиск выполнялся по ключевым словам и просмотром найденных мест.

## Найденные утечки и исправления

### 1) `packages/client/src/hooks/useNotification.ts`

**Проблема:**  
При показе уведомлений создавались:
- `setTimeout` для авто-закрытия,
- объекты `Notification` с обработчиками `onclick/onerror`.

Они не отслеживались централизованно и не очищались при размонтировании хука.  
Это могло накапливать таймеры и ссылки на callbacks/Notification.

**Исправление:**
- Добавлены `timeoutsRef` и `notificationsRef` для учёта всех активных таймеров/уведомлений.
- Добавлен cleanup в `useEffect(() => () => {...}, [])`:
  - `clearTimeout` для всех таймеров;
  - снятие обработчиков и `close()` для всех `Notification`.
- Логика повторного показа после `requestPermission` ограничена только случаем `granted`
  (исключён повторный вызов при `denied`).

**Результат:** ✅ утечка устранена.

---

### 2) `packages/client/src/main.tsx`

**Проблема:**  
Слушатель `window.load` для регистрации service worker добавлялся без `once/cleanup`.

**Исправление:**  
Слушатель переведён в одноразовый:
`window.addEventListener('load', registerSW, { once: true })`.

**Результат:** ✅ риск удержания лишнего обработчика убран.

---

### 3) `packages/client/src/components/IdleObserver/IdleObserver.tsx`

**Проблема:**  
После `clearInterval` сохранялась ссылка на таймер в `intervalId.current`.

**Исправление:**  
После очистки добавлено `intervalId.current = null`.

**Результат:** ✅ лишняя ссылка не удерживается.
